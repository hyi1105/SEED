# 怎麼跟 Agent 講 UI，少重複改

## 為什麼會一直改

畫面問題多半不是「寫錯」，而是**目標還沒被看見前無法鎖死**。沒看過成品，很難知道更好做法；看過之後目標常會變——這是學習，不是失敗。

昂貴的是：把「探索」當成「定稿」來打磨。

## 兩種模式（先分清楚）

| 模式 | 目的 | 產出應該長怎樣 | 你怎麼回 |
| --- | --- | --- | --- |
| **探索** | 讓大腦有東西可評 | 粗、可丟、只證明方向 | 「方向對／不對；下一版像 ____」 |
| **定稿** | 鎖死後收斂 | 可上線、細節到位 | 「只改 ____；其他別動」 |

沒看到之前用探索；看過且方向對了才定稿。混在一起就會變「一直微調很浪費」。

## 目標會變時，高手怎麼快速大改

1. **承認轉向，不要在舊殼上補丁**  
   方向錯了就整段重畫／重寫互動，比在舊 UI 上修十次便宜。
2. **一句話分「留／丟／新」**  
   留什麼資料與流程、丟什麼畫面、新隱喻是什麼。
3. **只改一層**  
   大改時這一輪只動結構或只動互動，不順便改配色文案。
4. **第一版要「便宜到敢丟」**  
   強者先要可點的骨架，不先要完美。看到才產生真需求。
5. **用「取代」不是「疊加」**  
   新需求出現時說「用 ____ 取代 ____」，避免功能越疊越亂。

## 探索階段（還沒看過／還在找感覺）

跟 Agent 說：

```text
這輪是探索，不要打磨。
目的：讓我看到一種做法。
假設：____
請只做最小可點版本；細節之後再說。
做完用三句說明：你選了什麼方向、我該看哪、哪裡可丟。
```

你回饋時只評方向：

```text
方向：對／偏了
我學到：____（看了才知道要的）
下一版隱喻：像 ____
請大改，不要在舊版上補丁
```

## 轉向大改（看完有新需求）

貼這段最快：

```text
【大改／轉向】
留：____（資料、流程、名稱）
丟：____（哪個畫面／互動整段刪）
新：像 ____（新隱喻或參考）
這一輪只做到：____ 可點就停
不要：微調舊版、順便改無關細節
```

## 定稿階段（方向已對）

```text
方向鎖定。請只改：____
其他別動。完成定義：____
```

## 跟本庫 Agent 的開場 brief（可删減）

```text
1. 模式：探索 / 定稿
2. 目標：使用者要完成____
3. 參考：像____
4. 必做：____
5. 不要：____
6. 完成定義：做到____就停
```

## 本專案已驗證

- 差：「編輯模板不直觀」→ 容易在舊表單上修文案  
- 好：「元件用拖移，如同新增表情符號」→ 一次轉向托盤拖放（大改對了）
- 真相：很多好想法是**看過第一版才出現**；重點是第一版要便宜、轉向要乾脆

## 決定／偏好

- 允許目標變；用「探索 → 轉向大改 → 定稿」三段，不假裝一開始就知道答案
- 大改時明確「留／丟／新」，不要在舊殼補丁
- 「像 XX」仍是最高槓桿溝通
